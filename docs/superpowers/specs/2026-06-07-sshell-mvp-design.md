# sshell MVP 设计规格

## 元信息

- 日期: 2026-06-07
- 主题: 终端兼容性优先的 sshell MVP
- 状态: 已评审，待实现计划

## 背景

sshell 的目标是提供一个使用 Rust 开发、可跨平台分发的 SSH 客户端命令行工具，既能由开发者直接在终端使用，也能被 Qt 应用作为独立进程启动。当前阶段聚焦 MVP，优先保证交互式终端兼容性，而不是扩展功能面。

现有文档已经定义了总体需求、模块划分和最小任务清单。本规格在这些草稿基础上补充并固化本轮确认过的产品决策，用于约束后续实现计划。

## 目标与非目标

### 目标

- 优先保证交互式终端兼容性。
- 明确支持 vim、top、less、htop 作为 MVP 核心验收程序。
- 支持通过命令行一次传入一个或多个连接串，启动多个独立 SSH 会话。
- 在单个 TUI 窗口内显示多 Tab，并允许切换和关闭。
- 对外提供稳定、可文档化的退出码，便于 Qt 侧按进程结果判断状态。

### 非目标

- 不支持运行时新建连接。
- 不支持密钥认证、会话保存、快速重连、文件传输、命令执行模式。
- 不提供 Qt 进程内控制协议或窗口嵌入能力。
- 不在 MVP 内引入图形界面或额外配置系统。

## 决策摘要

- 产品优先级: 终端兼容性优先于多 Tab 体验和 Qt 集成细节。
- 验收程序: vim、top、less、htop 必须可用。
- Tab 交互策略: 采用前缀键模式，默认键盘输入透传远端。
- 运行时建连策略: 仅支持启动时通过命令行传入连接串，MVP 不提供运行时输入框。

## 推荐方案

MVP 采用终端优先的一体式 TUI 方案。程序在单进程内完成 SSH 会话管理、终端仿真和 TUI 渲染，但通过清晰的内部模块边界隔离连接解析、会话管理、SSH 运行时、终端引擎和错误模型。

选择这一方案的原因有三点:

1. 它最贴合终端兼容性优先的目标，能把绝大多数键盘输入直接交给远端程序处理。
2. 它能在 MVP 阶段降低控制面复杂度，避免为了运行时建连或复杂快捷键牺牲终端行为正确性。
3. 它仍然保留后续演进空间，未来如果需要更深的 Qt 集成或更复杂的 UI，可以沿现有模块边界扩展，而不是整体推倒重来。

## 总体架构

系统遵循“远端终端优先，本地控制最小侵入”的原则。每个连接串对应一个独立 SSH 会话，每个会话拥有独立的 SSH 通道、终端状态和 Tab 标题。所有会话共享一个 TUI 外壳，但默认只有当前激活会话接收输入并渲染到主视图。

输入路径应尽量简单: 键盘事件先进入本地 TUI 外壳，若未命中前缀键控制流，则直接透传到当前 SSH 会话。输出路径同样保持单向清晰: 远端输出字节流进入 SSH 会话运行时，再喂给终端仿真引擎生成屏幕状态，最后由 TUI 渲染。窗口尺寸变化单独走 resize 通道，并同时同步到远端会话和本地终端缓冲。

该结构的关键点在于，直接决定终端兼容性的逻辑只应出现在 SSH I/O 链路和终端仿真引擎中，而不是散落在 Tab 或界面控制逻辑里。

## 组件设计

### 连接解析器

负责解析命令行中的 用户名:密码@主机[:端口]，并完成参数合法性校验。该模块只参与启动阶段，不携带运行时状态。格式错误统一归类为参数错误。

### 会话管理器

负责维护 Tab 顺序、当前激活会话、关闭后的相邻选择规则，以及“最后一个 Tab 关闭即退出”的生命周期判断。它不处理终端字节流和 SSH 协议细节，只维护会话元数据和调度状态。

### SSH 会话运行时

每个 Tab 对应一个独立运行时，负责建立连接、用户名密码认证、打开远端 shell、转发输入输出和同步窗口尺寸。它是终端兼容性的第一关键模块，因为远端程序是否能正常运行，首先依赖这里是否完整、无污染地传递字节流。

### 终端仿真引擎

负责解释 ANSI/VT 序列、维护屏幕缓冲、光标和 alternate screen 状态。它是终端兼容性的第二关键模块，因为 vim、top、less、htop 的正确显示本质上依赖这里能否准确重建屏幕状态。

### TUI 外壳

负责绘制 Tab 栏、会话视图和最小状态提示，同时执行前缀键状态机。它应保持足够薄，避免承担 SSH 或终端语义。默认情况下，绝大多数按键直接透传给远端会话，以减少与全屏程序的冲突。

### 错误模型与退出码聚合器

负责统一错误分类、stderr 输出格式和进程最终退出码决策。该模块既服务用户排障，也服务 Qt 调用方按退出码判断结果。

## 交互设计

MVP 使用前缀键模式，而不是直接快捷键模式。默认所有普通按键透传远端，仅在检测到前缀键后才将后续按键解释为本地控制命令。

建议保留与现有草稿一致的前缀方案，即使用 Ctrl+A 作为前缀键，并提供以下最小控制集:

- Ctrl+A, n: 切换到下一个 Tab
- Ctrl+A, p: 切换到上一个 Tab
- Ctrl+A, x: 关闭当前 Tab
- Ctrl+A, Ctrl+A: 发送字面 Ctrl+A 到远端

MVP 不提供运行时新建连接能力，因此不需要输入框或表单交互。多 Tab 仅由启动参数决定。

## 错误处理

错误处理分为会话内错误和进程级退出两层。

会话内错误只影响当前 Tab。例如认证失败、主机不可达、连接超时或远端异常关闭时，应在当前 Tab 显示清晰错误状态，但不自动终止其他仍然健康的会话。

进程级退出只在最后一个会话结束时决策。规则如下:

- 所有 Tab 由用户主动关闭，退出码为 0。
- 存在参数错误时，启动阶段直接以参数错误退出。
- 若全部会话均因异常结束，则返回首个异常退出码。
- 无法归类的错误回落到通用错误。

stderr 输出采用统一格式:

```text
[ERROR] <code>: <message>
```

## 退出码策略

沿用现有需求文档中的退出码定义:

- 0: 正常退出
- 1: 通用错误
- 2: 命令行参数错误
- 3: 网络错误
- 4: 认证失败
- 5: SSH 协议错误
- 6: 终端初始化失败

多会话场景下，退出码的意义是“整个进程在最后结束时给出的汇总结果”，而不是实时反映每个 Tab 的局部状态。

## 验收标准

### 终端兼容性验收

下列程序是 MVP 的强制验收基线:

- vim
- top
- less
- htop

每个程序至少应满足以下条件:

- 能正常启动并进入交互界面。
- 屏幕不会出现明显错乱或残留污染。
- 退出后能返回主 shell。
- 窗口 resize 后界面能正确更新。
- Tab 切换后返回该会话时，当前屏幕状态仍然正确。

### 生命周期验收

- 单连接成功时，能进入远端 shell 并正常交互。
- 错误口令时，当前 Tab 显示认证失败，进程退出语义符合规则。
- 主机不可达或连接超时时，错误归类为网络错误。
- 用户关闭单个异常 Tab 时，不影响其他 Tab 继续使用。
- 关闭最后一个 Tab 后，程序退出。

### Qt 集成验收

- Qt 可通过外部进程启动 sshell 并传入连接串。
- Qt 可通过进程退出码区分正常关闭与主要错误类型。
- Qt 无需了解内部 Tab 或会话状态，也无需参与窗口管理。

## 实施顺序

建议按以下顺序推进实现:

1. 打通单会话链路: 参数解析、SSH 连接、认证、远端 shell、输入输出转发、resize。
2. 稳定终端仿真与渲染: ANSI/VT、alternate screen、全屏程序退出恢复。
3. 加入多 Tab 管理与前缀键控制: 会话隔离、切换、关闭、退出决策。
4. 收敛错误模型和退出码聚合，并补 Qt 调用场景验证。

这个顺序的目标是先把最容易破坏用户体验的终端链路站稳，再叠加本地控制能力，而不是一开始就在复杂 UI 和多状态管理里排查问题。

## 风险与边界

- 如果 TUI 框架对原始键盘事件拦截过重，会直接损害全屏程序兼容性，需要优先验证。
- 如果终端仿真能力不足，可能出现 vim 或 htop 显示错乱，这类风险应在实现早期暴露。
- Windows ConPTY 兼容性仍是单独风险项，但不改变当前 MVP 的核心设计。
- 运行时不支持新建连接是刻意收缩范围，不应在实现过程中被“顺手加上”。

## 后续演进

未来若需要运行时新建连接、密钥认证、会话保存或 Qt 更深集成，优先沿现有模块边界扩展，而不是重写主循环。MVP 阶段的接口设计应默认未来会增长，但当前实现不提前承担这些功能成本。
